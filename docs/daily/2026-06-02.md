# 2026-06-02

## 会话摘要
修正对话工作台的可用性断点，补全多轮异步群聊，并把测试收敛到用户能实际依赖的程度。

## 记录
- [进展] 对话页修正 direct chat 参与者状态：单聊前会自动把选中人物加入当前会话，消除“0 位参与者但人物已回复”的状态错位。
- [进展] 群聊从一次性广播改为多轮异步 run：后端 `conversationRunService` 现按轮次推进、多人物逐轮发言、自动完成或手动终止，并持久化 `running/completed/stopped/failed` 状态。
- [进展] 前端 `ConversationWorkspace` 新增 run 轮询、参与者同步与运行态禁用控制；群聊进行中会刷新消息、参与者与 run 状态，不再只靠一次提交后的静态快照。
- [进展] 前端消息流补齐类型标签与长文本换行，人物状态 `needs_research` 显示改为中文 `待研究`，空会话提示明确区分单聊至少 1 人、群聊至少 2 人。
- [进展] 修复“加入会话 / 加入任务”真实报错：会话参与者表改为“每会话每人物唯一”，`skillId` 允许为空，后端自动解析最新 skill；真实人物即使尚未蒸馏 skill 也能先加入会话。
- [进展] 修复真实浏览器下“群聊状态进入 running 但消息不刷新”的断点：前端改为显式 run polling，不再依赖隐式 effect 定时器；群聊启动后立即拉取 messages/participants/run 状态，完成或终止后自动收口。
- [进展] 修复真实多人群聊被模型伪 JSON 打断的问题：聊天回复链改用纯文本完成接口，并在后端 prompt 中绑定当前 `personName`，避免一个人物替所有人发言。
- [经验] 对话产品最忌“消息已经发生、会话状态还停在旧世界”；参与者、消息、run 必须围绕同一个 conversation 真相源同步，否则 UI 看起来能用，语义却是错的。
- [经验] 群聊如果只是同步广播一轮，本质上仍是批量单聊；只有把 run 状态、轮次推进和 stop 写成后端编排，前端才有资格做“进行中 / 已停止 / 已完成”的用户体验。
- [经验] 聊天链路的测试不能只测按钮存在。至少要覆盖：单聊自动入场、群聊多人物启动、终止群聊、run 状态收敛，以及浏览器端真实点击录证。
- [经验] 前端不该伪造 `skillId`；如果参与者入场依赖前端猜一个快照 ID，外键迟早会炸。正确边界是后端按真实库自动选择最新 skill，没有就显式为空。
- [经验] 对话生成和结构化评审是两种完全不同的模型契约；评审继续要 JSON，聊天必须改成纯文本，否则真实群聊迟早会被引号、markdown fence 或伪 JSON 打断。
- [验证] `npm test` 通过 15 个测试文件共 46 个测试，覆盖对话工作台、会话运行服务、App seam 与后端路由。
- [验证] `npm run lint`、`npm run build`、`npm run test:e2e` 全部通过；Playwright 录证确认研究/蒸馏/对话三页仍可点击，完整对话流可跑通“加人→单聊→群聊→终止群聊”。
- [验证] 真实后端链路：直接请求 `POST /api/conversations/:id/participants` 且不传 `skillId`，当前返回 `201` 与 `skillId: null`，不再出现 `FOREIGN KEY constraint failed`。
- [验证] 真实浏览器链路：单聊页出现 `2` 张消息卡（用户 + Paul Graham 回复）；多人群聊页出现 `6` 张消息卡并含 `Paul Graham` 与 `张一鸣` 两位人物回复；再次拉起群聊后点击 `终止群聊`，页面进入 `群聊已停止` 且历史保留。

## Retain
- → bank/experience: 对话工作台的参与者、消息、run 必须围绕同一 `conversationId` 同步；单聊自动入场能消除“有人发言但无人入场”的状态错位。
- → bank/experience: 真正的群聊需要后端多轮异步编排与可停止的 run 状态；一次性广播只是假群聊。
- → bank/experience: 聊天产品测试至少要同时覆盖服务层 run 状态机、前端 seam 与浏览器端真实点击录证。
- → bank/experience: 前端不要伪造 skill 快照 ID；参与者入会话应由后端自动解析最新 skill，没有就允许 `skillId = null`，否则真实人物库一接入就会被外键打爆。
- → bank/experience: 聊天链路不要复用 JSON 评审契约；真实多人群聊应使用按 speaker 约束的纯文本回复，并由前端显式 polling 同步 run 状态与消息流。
