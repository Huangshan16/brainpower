## Retain
- → MEMORY: 分表、分服务、分 UI 面板是本项目核心约束。
- → bank/experience: 数字人物输出必须带证据、置信度和诚实边界，避免变成无来源拟人表演。
- → bank/world: SQLite 基础 schema 固定七张核心表：people、sources、fragments、skills、evaluations、critiques、jobs。
- → bank/experience: 若 Vitest setupFiles 指向缺失文件，业务测试会在收集前失败；先恢复测试入口再验证目标红测。
- → bank/experience: 后端路由保持薄层，SQL 只进入 service 文件；测试用 shared schema parse 防止数据库字段命名漂移。
- → bank/experience: researchService 用 URL 去重 + 正文 hash 去重的双重约束，比单纯堆更多 crawl_status 分支更稳。
- → bank/experience: 对评审记录这类快照数据，硬外键会把测试和演进都绑死；保留不可替代关系的外键，人物/技能引用改成快照字段更稳。
- → bank/experience: 前端 seed 数据若用 `as const` 直接喂给可变组件 props，TypeScript 很容易在 `readonly` 和字面量状态上绊住 build；工作台 seed 更适合显式宽类型。
- → bank/experience: 前端纵向切片验收先补 API seam 测试，再用独立浏览器截图确认桌面三列与窄视口单列；这样能把逻辑回归和布局回归分开验证。
- → bank/experience: Playwright 浏览器录证要先把 locator 锁进明确容器，再让 Vitest/Playwright 各自只收自己的测试文件；否则测试边界会互相污染。
- → bank/experience: 开源 README 必须写清“已实现什么、未实现什么、如何启动、如何验证、如何贡献”，否则首页会把原型项目伪装成成品。
